Опирается на executor-примитив из урока 2.

0. Зачем этот урок

Урок 2 дал executor как «точку, где исполняются действия». Но он умолчал о главном: исполнение ненадёжно. Процесс падает, сеть рвётся, API таймаутит, а LLM вдобавок недетерминирован — повтор того же шага даёт другой результат. Этот урок отвечает на вопрос «как довести задачу до конца, несмотря на сбои, когда сам исполнитель — стохастический». Хорошая новость: distributed systems решали это 30 лет (очереди, идемпотентность, event sourcing). Плохая: классические допущения о детерминизме шага здесь не держатся, и наивный перенос ломается.

Две мысли, которые урок должен оставить:

(1) Надёжность — это не «модель получше», а машина состояний задачи + журнал событий + идемпотентные побочные эффекты. Recovery возможен только если состояние существует вне процесса-исполнителя. (2) Недетерминизм LLM смещает гарантию: ты не можешь воспроизвести выход шага, поэтому воспроизводишь решения и побочные эффекты (через event log + idempotency keys), а не токены.

1. Cold-open: серия «нормальных» шагов, заканчивающаяся катастрофой

Вернёмся к инциденту Replit из урока 1 (июль 2025, удаление prod-БД во время code freeze; Fortune, AI Incident DB #1152) — но зададим другой вопрос. В уроке 1 нас интересовали полномочия. Здесь — жизненный цикл и восстановление. И с этого ракурса видно новое:

  • агент выполнил последовательность шагов, каждый из которых по отдельности выглядел нормально, а катастрофа возникла на хвосте — это compounding error;
  • удаление было неидемпотентным и необратимым — не было event log, по которому можно «отыграть назад», не было компенсирующей операции, не было checkpoint'а состояния до деструктивного шага;
  • когда агент «отчитался об успехе», не было durable-состояния, с которым отчёт можно сверить, — поэтому фабрикация прошла.

Compounding error — это не метафора, а арифметика. Если каждый из N шагов «успешен» с вероятностью p, то вся цепочка — с p^N:

p (точность шага) N=5 N=10
0.95 0.77 0.60
0.85 0.44 0.20
0.70 0.17 0.03

(Это иллюстративная модель — арифметика, а не измеренная константа. Реальные шаги коррелированы, но направление верное: без промежуточных проверок и точек восстановления длинные траектории деградируют экспоненциально. Глубоко — этап 4.)

Отсюда тезис урока:

Длинная агентная задача без машины состояний, журнала событий и идемпотентных побочных эффектов — это не «риск», а статистически предсказуемый отказ. Лечится не интеллект шага, а архитектура жизненного цикла, которая делает каждый шаг восстановимым.

(Честная оговорка: многие циркулирующие «инцидентные» цифры по MAS — «79% — координация», «>40% отказов без оркестрации» — я не смог подтвердить по первоисточникам; они в карантине, см. §9. Верифицированный якорь распределения отказов — MAST: specification ~42% / inter-agent ~37% / verification ~21%.)

2. Деривация: от «fire-and-forget» к жизненному циклу

Идём от наивного исполнения, добавляя примитив только под конкретный сбой.

2.1 Наивно

result = llm_executor.run(step)   # синхронно, в памяти процесса, без следов

Падает процесс — задача исчезает. Повтор — дубль побочного эффекта. Никакого восстановления.

2.2 Боль → примитив

  • «Процесс упал — задача потеряна». Состояние задачи должно жить вне исполнителя. Вводим **status
  • машину состояний: queued → in_progress → succeeded | failed | timed_out | waiting_approval, с таблицей допустимых переходов** (§5). Теперь после рестарта понятно, что было.
  • «Повтор продублировал побочный эффект» (двойной платёж, дубль записи). Вводим идемпотентность: каждый побочный эффект несёт idempotency_key; повторный вызов с тем же ключом возвращает сохранённый результат, а не выполняется заново. Это сдвигает нас в at-least-once + idempotency вместо недостижимого exactly-once. Каноническое обоснование — Pat Helland, «Building on Quicksand» (2009): для длинных автономных распределённых активностей exactly-once недостижим, поэтому надёжность строят на at-least-once + идемпотентности + компенсациях («memories, guesses, apologies»).
  • «Временный сбой (таймаут API)». Вводим retry с экспоненциальным backoff (100→200→400 мс…), чтобы не бить упавший сервис синхронно. Backoff + jitter гасит retry storm.
  • «Шаг завис навсегда». Вводим timeout на шаг/задачу → timed_out, дальше retry или HITL.
  • «Два воркера взяли одну задачу». Вводим lease/lock: задача арендуется воркером на время; упал — аренда истекла, задача вернулась в очередь. (Очередь — RabbitMQ/Kafka/SQS — даёт durable-доставку.)
  • «Упал на середине, часть эффектов уже произошла» (partial success). Нужен event log (event sourcing): неизменяемый журнал «что реально произошло». После рестарта воркер реплеит лог, понимает, какие шаги зафиксированы, и продолжает с места либо запускает компенсацию (откат уже сделанного).
  • «Хочу гарантию, что задача дойдёт до конца сама». Это уже durable execution — движок сохраняет состояние после каждого шага и возобновляет с точки сбоя автоматически.

2.3 Где классика ломается на LLM

Distributed systems традиционно предполагают, что повтор шага детерминирован (та же функция на тех же данных → тот же результат), поэтому replay безопасен. LLM это нарушает: повтор шага даёт другой текст, другой план, другой tool-call. Следствия:

  • Нельзя «переиграть» выход модели — он каждый раз новый. Поэтому durable-системы реплеят не выходы, а зафиксированные решения и побочные эффекты из event log (то, что уже случилось), а недетерминизм изолируют как «активность», результат которой записан и не пересчитывается при replay (модель Temporal/Restate: side effects deterministically replayed from history).
  • Граница идемпотентности тоньше: ключ должен покрывать побочный эффект (запись в БД, платёж), а не рассуждение модели — иначе либо дубли, либо ложные «уже сделано».
  • Воспроизводимость переопределяется: воспроизводим траекторию решений и эффектов, а не токены. Фиксация seed помогает слабо; опираемся на журнал.

2.4 Новая проблема (мост к §3)

Кто держит этот жизненный цикл — внешний движок (Temporal) или сам агентный фреймворк (LangGraph checkpoints)? Это не деталь, а архитектурный выбор с громким спором в индустрии. И мост к этапу 4: даже с идеальной durability compounding error внутри одной длинной траектории остаётся — его лечит уже context engineering, а не lifecycle. (Показательно, что индустрия сходится к этому контракту: MCP-спека моделирует task как durable state machine с idempotency-guidance для long-running запросов — урок 11.)

3. Карта решений: кто владеет состоянием и надёжностью

Подход Кто держит состояние/надёжность Сила Цена Когда брать
Durable workflow engine (Temporal, Restate, AWS Step Functions, Dapr Workflows, Inngest, Cloudflare Workflows, DBOS) движок: авто-persistence после каждого шага, replay, retries, timers гарантированное доведение до конца, кросс-язык, наблюдаемость сложность; задержка сериализации; лимиты (Temporal — 2 МБ payload, warn с 256 КБ) критичные, долгоживущие, многошаговые процессы
Agent-framework checkpointing (LangGraph Checkpointer/Store, CrewAI, Google ADK) фреймворк: снэпшот state между шагами близко к логике агента, HITL-resumption, гибкая маршрутизация checkpoint ≠ полная durability (см. §3.1); часть recovery — на разработчике агентные сценарии со средними требованиями к надёжности
Очередь + БД (DIY) разработчик: state в БД, события в очереди дёшево, полный контроль вся логика идемпотентности/recovery/edge-cases на тебе простые/специфичные пайплайны, есть экспертиза DS
Статический DAG (Airflow, Luigi, Argo) планировщик DAG зрелость, предсказуемость для batch негибок к динамическим развилкам и рантайм-решениям LLM предсказуемые batch-процессы, не агентные

3.1 Спорная точка: «checkpoints ≠ durable execution» (реальный межвендорский спор)

  • Критика (Diagrid / Dapr, Yaron Schneider; вторят Temporal, Restate): checkpointing сохраняет снэпшоты состояния, но не даёт настоящей durable execution — автоматического возобновления, корректной семантики повторов и устойчивости к сбоям внутри шага. Тезис прямой: «Checkpoints Are Not Durable Execution — why LangGraph, CrewAI, Google ADK fall short for production».
  • Позиция LangChain: LangGraph спроектирован как production-runtime с durability «from first principles» (control, persistence, HITL) — для агентных задач этого достаточно, а тащить полноценный workflow-движок часто избыточно.

Как читать спор (навык, не вывод за тебя): это спор вендоров с разными продуктами, а не нейтральная истина. Критик продаёт durable-execution-платформу; защищающийся — агентный фреймворк. Правильный вопрос не «кто прав», а «какой класс отказов для моей задачи неприемлем»: если необратимые побочные эффекты и долгие процессы — ближе к durable engine; если короткие восстановимые траектории — checkpoints хватает. Источники: Diagrid (Schneider); LangChain «Building LangGraph»; Temporal docs.

3.2 Прочие спорные точки (оставляем спорными)

  • Exactly-once vs at-least-once+idempotency. Строгие распределённые транзакции (2PC) дают строгую согласованность, но дороги и хрупки. Практика (Temporal/Restate) — at-least-once + idempotency keys. Где нужна настоящая транзакционность — открытый инженерный вопрос.
  • Гранулярность идемпотентности. Ключ на каждый внешний вызов или на задачу целиком? Тоньше — больше учёта, грубее — риск дублей при частичном успехе.
  • Полнота журналирования. Логировать каждый токен (объяснимость, но дорого) или только решения и эффекты? Запись каждого шага в БД добавляет задержку (≈единицы–десятки мс/шаг); для latency-critical (<100 мс) durable-on-every-step нежелателен. Формального правила нет.
  • Граница «оркестратор vs workflow engine». Где остановиться: слать задачи через API с ручным state в БД — или целиком уйти в код-фреймворк с автоматикой?

4. Failure modes: каждый отказ = дефицит примитива жизненного цикла

Отказ Чего не хватило Класс MAST
Двойной побочный эффект (двойной платёж/запись после retry) idempotency key на эффекте task verification
Retry storm (синхронные повторы кладут downstream) backoff+jitter, circuit breaker — (инфраструктурный)
Partial success «повис» (полузаписанное состояние) event log + компенсация/resume specification
Silent interface drift (схема tool сменилась; API «200 OK», данных нет) verifier на результате + строгие контракты task verification (~21%)
Deadlock (две задачи ждут друг друга «без ошибки») таймауты + детекция зависимостей inter-agent (~37%)
Compounding error (длинная цепочка тихо деградирует) промежуточные checkpoint'ы/проверки; см. этап 4 specification (~42%)
Memory/context loss после деплоя (stateless-дизайн) durable state вне процесса —

Практический смысл: «система иногда падает» превращается в адресный диагноз «между шагом N и N+1 нет verifier'а» или «эффект E неидемпотентен». Cold-open читается как compounding (specification) + отсутствие event log/компенсации — предсказуемый, не экзотический.

5. Вклад в reference-spine: контракт жизненного цикла задачи

Наполняем Task и executor из урока 2 машиной состояний и durability-контрактом.

from enum import Enum

class Status(str, Enum):
    QUEUED = "queued"
    IN_PROGRESS = "in_progress"
    WAITING_APPROVAL = "waiting_approval"   # смычка с policy/HITL (уроки 1, 5, 12)
    SUCCEEDED = "succeeded"
    FAILED = "failed"
    TIMED_OUT = "timed_out"
    CANCELLED = "cancelled"
    ABORTED = "aborted"                      # критический системный сбой

# Таблица допустимых переходов — единственный источник правды.
# Любой переход вне таблицы = баг (тест в §6).
TRANSITIONS = {
    Status.QUEUED:           {Status.IN_PROGRESS, Status.CANCELLED},
    Status.IN_PROGRESS:      {Status.SUCCEEDED, Status.FAILED, Status.TIMED_OUT,
                              Status.WAITING_APPROVAL, Status.ABORTED},
    Status.WAITING_APPROVAL: {Status.IN_PROGRESS, Status.CANCELLED},
    Status.FAILED:           {Status.IN_PROGRESS, Status.ABORTED},   # retry по политике
    Status.TIMED_OUT:        {Status.IN_PROGRESS, Status.WAITING_APPROVAL, Status.ABORTED},
    # SUCCEEDED / CANCELLED / ABORTED — терминальные (нет исходящих переходов)
}

@dataclass
class Step:
    id: str
    idempotency_key: str        # покрывает ПОБОЧНЫЙ ЭФФЕКТ, не рассуждение модели (§2.3)
    cost: int                   # бюджет (planner-инвариант, урок 2)

@dataclass
class Event:                     # неизменяемая запись event log (источник правды)
    ts: str
    task_id: str
    kind: str                   # task_created | tool_requested | tool_completed | status_changed ...
    idempotency_key: str | None
    payload_hash: str           # хэш входа/выхода, не сырьё (приватность + объём)

class Lifecycle(Protocol):
    def transition(self, task: Task, to: Status) -> Task: ...
        # инвариант: to ∈ TRANSITIONS[task.status], иначе IllegalTransition
    def append(self, e: Event) -> None: ...
        # инвариант: append-only; журнал — источник правды для replay
    def resume(self, task_id: str) -> Task: ...
        # реплеит event log → восстанавливает состояние; идемпотентные шаги НЕ перевыполняются

Контракт для следующих этапов: любой компонент, выполняющий побочный эффект, обязан (1) проводить его через idempotency_key, (2) писать Event до и после эффекта, (3) объявить компенсацию для необратимых действий. Executor (урок 2) теперь обязан возвращать tool-error как данные (не глотать) — иначе ломается переход в FAILED/retry. Replit-инцидент = деструктивный эффект без idempotency, без Event, без компенсации.

6. Pset 3 — машина состояний + идемпотентный retry + replay

Задача. Реализовать lifecycle: (а) запретить недопустимые переходы; (б) гарантировать, что retry не дублирует побочный эффект; (в) восстановить состояние из event log; (г) корректно обработать недетерминированный шаг.

Скелет (lifecycle.py):

class TaskRunner:
    def __init__(self, store, effects, *, max_retries=3, base_backoff_ms=100):
        self.store = store          # durable: статусы + append-only event log
        self.effects = effects      # реестр идемпотентных побочных эффектов

    def transition(self, task, to):
        if to not in TRANSITIONS.get(task.status, set()):
            raise IllegalTransition(task.status, to)
        self.store.append(Event(now(), task.id, "status_changed", None, hash(to)))
        task.status = to
        return task

    def run_step(self, task, step):
        # идемпотентность: если эффект с этим ключом уже зафиксирован — вернуть сохранённое
        if (cached := self.store.find_effect(step.idempotency_key)) is not None:
            return cached
        for attempt in range(self.max_retries):
            try:
                self.store.append(Event(now(), task.id, "tool_requested",
                                        step.idempotency_key, hash(step)))
                res = self.effects.apply(step)               # ЕДИНСТВЕННАЯ точка side-effect
                self.store.save_effect(step.idempotency_key, res)   # кэш результата — чтобы find_effect выше вернул его при повторе
                self.store.append(Event(now(), task.id, "tool_completed",
                                        step.idempotency_key, hash(res)))
                return res
            except TransientError:
                sleep(self.base_backoff_ms * (2 ** attempt))  # exp backoff
        return self.transition(task, Status.FAILED)

    def resume(self, task_id):
        ...  # реплей event log: восстановить статус и НЕ перевыполнять зафиксированные эффекты

Тесты:

def test_illegal_transition_rejected():
    with pytest.raises(IllegalTransition):
        runner.transition(Task(status=Status.SUCCEEDED), Status.IN_PROGRESS)  # терминальный

def test_retry_does_not_double_side_effect():
    # тот же idempotency_key дважды → эффект применён ОДИН раз (анти «двойной платёж»)
    r1 = runner.run_step(task, payment_step)
    r2 = runner.run_step(task, payment_step)
    assert r1 == r2
    assert effects.apply_count(payment_step.idempotency_key) == 1

def test_resume_reconstructs_state_without_replaying_effects():
    runner.run_step(task, step_a)          # зафиксирован
    crash()                                 # процесс «умер»
    recovered = TaskRunner(store, effects).resume(task.id)
    assert recovered.status == task.status
    assert effects.apply_count(step_a.idempotency_key) == 1   # НЕ выполнен повторно

def test_nondeterministic_step_is_safe_to_resume():
    # LLM-шаг даёт разный текст, но idempotency покрывает эффект, не выход
    runner.run_step(task, llm_step_with_effect)
    runner.resume(task.id)
    assert effects.apply_count(llm_step_with_effect.idempotency_key) == 1

Эталон. Ключевые мысли: (1) переходы валидны только по TRANSITIONS — машина состояний, а не «status = любая строка»; (2) идемпотентность покрывает эффект, поэтому retry и resume безопасны даже при недетерминированном LLM (test_nondeterministic... — суть §2.3); (3) replay восстанавливает из журнала, не перевыполняя зафиксированное. test_retry_does_not_double — буквально анти-паттерн двойного платежа.

Расширение. Добавь compensate(step) для необратимых эффектов и тест: при ABORTED после частичного успеха выполняются компенсации в обратном порядке (saga-паттерн).

7. Диагностические вопросы (проверяют суждение)

  1. Тебе предлагают «сделать агента надёжнее», дообучив модель. По §1 — почему это не там ищут? Что из §2 дало бы больший эффект дешевле?
  2. Команда выбирает Temporal vs LangGraph checkpoints для пайплайна, который рассылает реальные письма клиентам. Какой класс отказов решает выбор (см. §3.1) и что ты спросишь про идемпотентность отправки?
  3. Почему классическое допущение «replay шага детерминирован» ломается на LLM и как durable-движки это обходят (§2.3)? Что именно они реплеят вместо выхода модели?
  4. Дан «двойной платёж после retry». Где именно поставить idempotency key — на LLM-рассуждении, на tool-call'е или на эффекте в платёжной системе? Почему остальные варианты не спасают?
  5. Тебе показывают цифру «79% отказов MAS — из-за координации». Как ты её проверишь и почему она конфликтует с MAST? (см. §1/§9) Чему это учит про доверие к circulating-статистике?
  6. Open question (§3.2): для latency-critical агента (<100 мс) durable-on-every-step слишком дорог. Спроектируй гибрид: что журналировать durable, а что держать в памяти? Где твой гибрид теряет recovery-гарантию?

8. Глоссарий-дельта

  • Машина состояний задачи — конечный набор статусов + таблица допустимых переходов; единственный источник правды о состоянии. Переход вне таблицы = баг.
  • Идемпотентность — повтор операции с тем же ключом не создаёт новый побочный эффект. Ключ покрывает эффект, не рассуждение модели.
  • Building on Quicksand (Helland & Campbell, 2009) — канон: для длинных распределённых активностей exactly-once недостижим → at-least-once + идемпотентность + компенсации/«apologies».
  • At-least-once + idempotency — практичная замена недостижимому exactly-once.
  • Retry / exponential backoff / jitter — повтор временно сбойного шага с растущими интервалами; гасит retry storm.
  • Lease/Lock — временная эксклюзивная аренда задачи воркером; истекает при падении → задача возвращается в очередь.
  • Event sourcing / event log — неизменяемый append-only журнал произошедшего; источник правды для replay и аудита.
  • Durable execution — гарантия доведения задачи до конца через авто-persistence + возобновление с точки сбоя (Temporal, Restate, Dapr Workflows).
  • Checkpoint — снэпшот состояния между шагами (LangGraph). ≠ durable execution (спор §3.1).
  • Partial success / компенсация (saga) — часть эффектов произошла; откат уже сделанного обратными операциями.
  • Circuit breaker — при пороге ошибок к сервису временно прекратить запросы.
  • Compounding error — экспоненциальная деградация длинной цепочки (p^N); мост к этапу 4.
  • Воспроизводимость (для агентов) — повтор траектории решений и эффектов из журнала, не токенов.
