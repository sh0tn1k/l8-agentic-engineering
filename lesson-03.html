0. Зачем этот урок КОНТЕКСТ

Урок 2 дал исполнитель (executor) как «точку, где исполняются действия». Но он умолчал о главном: исполнение ненадёжно. Процесс падает, сеть рвётся, API таймаутит, а LLM вдобавок недетерминирован — повтор того же шага даёт другой результат. Этот урок отвечает на вопрос «как довести задачу до конца, несмотря на сбои, когда сам исполнитель — стохастический». Хорошая новость: распределённые системы решали это 30 лет (очереди, идемпотентность, журнал событий). Плохая: классические допущения о детерминизме шага здесь не держатся, и наивный перенос ломается.

Две мысли, которые урок должен оставить:

КЛЮЧЕВАЯ МЫСЛЬ

(1) Надёжность — это не «модель получше», а машина состояний задачи + журнал событий + идемпотентные побочные эффекты. Восстановление возможно только если состояние существует вне процесса-исполнителя. (2) Недетерминизм LLM смещает гарантию: ты не можешь воспроизвести выход шага, поэтому воспроизводишь решения и побочные эффекты (через журнал событий + ключи идемпотентности), а не токены.

1. Конкретная проблема: почему длинная траектория — статистически предсказуемый отказ КОНТЕКСТ

Вернёмся к инциденту Replit (урок 1, июль 2025) — но зададим другой вопрос. В уроке 1 нас интересовали полномочия. Здесь — жизненный цикл и восстановление. С этого ракурса видно новое:

  • агент выполнил последовательность шагов, каждый из которых по отдельности выглядел нормально, а катастрофа возникла на хвосте — это накопление ошибок (compounding error);
  • удаление было неидемпотентным и необратимым — не было журнала событий, по которому можно «отыграть назад», не было контрольной точки состояния до деструктивного шага;
  • когда агент «отчитался об успехе», не было устойчивого состояния, с которым отчёт можно сверить, — фабрикация прошла.

Начинаем с наивного

Вот как выглядит «исполнение» в простейшем агенте:

result = llm_executor.run(step)   # синхронно, в памяти процесса, без следов

Падает процесс — задача исчезает. Повтор — дубль побочного эффекта. Никакого восстановления.

Накопление ошибок — не метафора, а арифметика. Если каждый шаг «успешен» с вероятностью p, вся цепочка из N шагов — с вероятностью p^N:

p (точность шага) N=5 N=10
0.95 0.77 0.60
0.85 0.44 0.20
0.70 0.17 0.03

(Иллюстративная модель, не измеренная константа. Реальные шаги коррелированы, но направление верное: без промежуточных проверок длинные траектории деградируют экспоненциально.)

Отсюда тезис урока:

Длинная агентная задача без машины состояний, журнала событий и идемпотентных побочных эффектов — это статистически предсказуемый отказ. Лечится не интеллект шага, а архитектура жизненного цикла, которая делает каждый шаг восстановимым.

Разберём примитивы надёжности по одному — каждый как ответ на конкретный сбой из этого сценария.

2. Первая пара: повтор и идемпотентность — когда временный сбой не должен плодить дубли

Проблема: временный сбой + повтор = двойной эффект

Представьте: агент делает платёж. API вернуло timeout, но на самом деле платёж прошёл (распространённый случай в распределённых системах — потеря подтверждения). Агент повторяет шаг — платёж проведён дважды.

def naive_run(step):
    return call_external_api(step)   # timeout → повтор → double charge

Решение: идемпотентность

Идея: каждый побочный эффект несёт уникальный ключ идемпотентности. Повтор с тем же ключом возвращает сохранённый результат, а не выполняет эффект заново. Это сдвигает нас в at-least-once + идемпотентность вместо недостижимого exactly-once.

Каноническое обоснование — Pat Helland, «Building on Quicksand» (2009): для длинных распределённых активностей exactly-once недостижим, поэтому надёжность строят на at-least-once + идемпотентности + компенсациях.

def run_step(step, idempotency_key: str) -> Result:
    # Если эффект с этим ключом уже зафиксирован — вернуть сохранённое
    if (cached := store.find_result(idempotency_key)) is not None:
        return cached

    # Первое исполнение: выполнить, сохранить, вернуть
    result = execute_side_effect(step)
    store.save_result(idempotency_key, result)
    return result

Добавляем повтор с отсрочкой

Одна идемпотентность не решает временные сбои. Нужен повтор (retry) с экспоненциальной отсрочкой (backoff) — 100 → 200 → 400 мс — чтобы не бить упавший сервис синхронно.

def run_step_with_retry(step, idempotency_key, max_retries=3, base_backoff_ms=100):
    for attempt in range(max_retries):
        try:
            return run_step(step, idempotency_key)
        except TransientError:
            if attempt == max_retries - 1:
                raise Failed("step failed after all retries")
            sleep(base_backoff_ms * (2 ** attempt))  # exp backoff

Что изменилось

Повтор больше не создаёт дубли: ключ идемпотентности гарантирует, что каждый побочный эффект выполняется ровно один раз (в терминах наблюдаемого результата). Отсрочка гасит шторм повторов (retry storm).

Ограничение

Повтор не поможет, если шаг завис навсегда (бесконечное рассуждение LLM, зависший API). Нужен таймаут. И если два воркера взяли одну задачу — нужна аренда.

3. Вторая пара: таймаут и аренда — когда шаг завис или задачу взяли двое

Проблема: шаг завис навсегда

LLM может войти в бесконечное рассуждение, API может не ответить. Без таймаута система зависает на одном шаге навсегда.

Решение: таймаут

def run_step_with_timeout(step, idempotency_key, timeout=30):
    try:
        return run_step_with_retry(step, idempotency_key)
    except TimeoutError:
        transition(task, Status.TIMED_OUT)   # шаг помечен как таймаут
        return handle_timeout(task, step)    # повтор или HITL

Таймаут превращает «завис навсегда» в предсказуемый переход состояния: IN_PROGRESS → TIMED_OUT.

Проблема: два воркера взяли одну задачу

В распределённой системе два экземпляра агента могут одновременно взять одну задачу из очереди. Каждый выполнит шаг — дважды. Идемпотентность спасёт эффект, но двойная работа жжёт токены (помните ≈15× из урока 2?).

Решение: аренда

Аренда (lease/lock): задача временно закрепляется за воркером. Пока аренда жива — другие воркеры не трогают задачу. Если воркер упал, аренда истекает, и задача возвращается в очередь.

def acquire_lease(task_id: str, ttl_seconds=60) -> bool:
    """Попытаться захватить задачу. True — захвачено, False — уже у другого."""
    return redis.setnx(f"lease:{task_id}", ttl_seconds)  # атомарно, если нет — False

def release_lease(task_id: str):
    redis.delete(f"lease:{task_id}")

def process_task(task):
    if not acquire_lease(task.id):
        return  # задача уже у другого воркера
    try:
        for step in task.steps:
            run_step_with_timeout(step, ...)
    finally:
        release_lease(task.id)

Что изменилось

Таймаут не даёт шагу зависнуть навсегда. Аренда не даёт двум воркерам жечь токены на одной задаче. Вместе они делают исполнение предсказуемым по времени и ресурсам.

Ограничение

Что если воркер упал на середине задачи? Первые шаги выполнены, их побочные эффекты уже произошли, а состояние потеряно. Таймаут и аренда не восстанавливают частичный успех.

4. Третья пара: журнал событий и устойчивое исполнение — когда упал на середине

Проблема: частичный успех

Вернёмся к Replit. Представьте, что перед удалением базы агент выполнил три шага подготовки: прочитал схему, сгенерировал миграцию, запустил тесты. Все три — успешно. На четвёртом (удаление) процесс упал. После рестарта:

  • статус неизвестен (был в памяти процесса);
  • три успешных эффекта не откатились;
  • четвёртый шаг невозможно продолжить — нет записи, на чём остановились.

Решение: журнал событий

Вводим неизменяемый append-only журнал (event sourcing): каждое существенное событие (начало шага, результат, решение) записывается до того, как произойдёт следующий шаг. Журнал — единственный источник правды.

def run_step(task, step, idempotency_key):
    # 1. Записать «начинаю шаг» ДО эффекта
    store.append(Event(
        kind="tool_requested",
        task_id=task.id,
        idempotency_key=idempotency_key,
        payload=serialize(step)
    ))

    # 2. Выполнить (идемпотентно)
    result = execute_with_retry(step, idempotency_key)

    # 3. Записать «шаг завершён»
    store.append(Event(
        kind="tool_completed",
        task_id=task.id,
        idempotency_key=idempotency_key,
        payload=serialize(result)
    ))

    return result

После падения новый воркер может восстановить состояние по журналу:

def resume(task_id):
    events = store.read_events(task_id)
    for e in events:
        if e.kind == "tool_completed":
            continue   # уже сделано — не перевыполнять
        if e.kind == "tool_requested":
            # сбой произошёл на этом шаге — продолжаем с него
            step = deserialize(e.payload)
            return run_step(task, step, e.idempotency_key)
    return task.status

Новая проблема: LLM ломает классический replay

В распределённых системах есть негласное допущение: повтор шага детерминирован — та же функция на тех же данных даёт тот же результат. Поэтому replay безопасен: можно «переиграть» весь workflow, и получится то же самое.

LLM это нарушает. Повтор шага даёт другой текст, другой план, другой tool-call. Если при replay перевызвать LLM — получится другое рассуждение, другие решения, другой результат. Это не баг — это фундаментальное свойство стохастической модели.

Решение: replay эффектов, а не токенов

Устойчивые движки (Temporal, Restate) обходят это так: они реплеят не выход модели, а зафиксированные решения и побочные эффекты из журнала событий. То, что уже случилось, записано и не пересчитывается. Недетерминизм изолирован внутри шага: вызвали LLM → записали результат → при replay не вызываем LLM заново, а читаем запись.

def deterministic_replay(step, history):
    # При восстановлении: не вызываем LLM заново
    # Читаем зафиксированный результат из журнала
    if (recorded := store.find_result(step.idempotency_key)) is not None:
        return recorded
    # Только если шаг НЕ был выполнен — выполняем
    return execute_with_retry(step, step.idempotency_key)

Что изменилось

Журнал событий даёт возможность восстановиться после падения на любом шаге. Устойчивое исполнение (durable execution) гарантирует доведение до конца через авто-persistence + replay. Недетерминизм LLM не проблема, потому что replay опирается на зафиксированные эффекты, а не на перевызов модели.

Граница примитивов этого урока

Даже с идеальной устойчивостью накопление ошибок внутри одной длинной траектории остаётся — качество тихо деградирует от шага к шагу. Это лечит уже контекстная инженерия (урок 4), а не жизненный цикл.

Движок устойчивого исполнения (Temporal, Restate, AWS Step Functions, Dapr Workflows, Inngest, Cloudflare Workflows, DBOS) движок: авто-persistence после каждого шага, replay, retries, timers гарантированное доведение до конца, кросс-язык, наблюдаемость сложность; задержка сериализации; лимиты (Temporal — 2 МБ payload, warn с 256 КБ) критичные, долгоживущие, многошаговые процессы

Очередь + БД (DIY) разработчик: state в БД, события в очереди дёшево, полный контроль вся логика идемпотентности/recovery/пограничных случаев на тебе простые/специфичные пайплайны, есть экспертиза DS
Статический DAG (Airflow, Luigi, Argo) планировщик DAG зрелость, предсказуемость для batch негибок к динамическим развилкам и рантайм-решениям LLM предсказуемые batch-процессы, не агентные

Спорная точка: контрольные точки ≠ устойчивое исполнение

Это реальный межвендорский спор. Diagrid / Dapr (Yaron Schneider): контрольные точки сохраняют снэпшоты состояния, но не дают настоящего устойчивого исполнения — автоматического возобновления, корректной семантики повторов, устойчивости к сбоям внутри шага. LangChain: LangGraph спроектирован как production-runtime с устойчивостью «from first principles» — для агентных задач этого достаточно.

Как читать спор: это спор вендоров с разными продуктами, а не нейтральная истина. Правильный вопрос не «кто прав», а «какой класс отказов для моей задачи неприемлем»: если необратимые побочные эффекты и долгие процессы — ближе к устойчивому движку; если короткие восстановимые траектории — контрольных точек хватает.

Прочие спорные точки: - Exactly-once vs at-least-once + идемпотентность. Строгие распределённые транзакции (2PC) дают согласованность, но дороги и хрупки. Практика (Temporal/Restate) — at-least-once + ключи идемпотентности. - Гранулярность идемпотентности. Ключ на каждый внешний вызов или на задачу целиком? Тоньше — больше учёта, грубее — риск дублей при частичном успехе. - Полнота журналирования. Логировать каждый токен (объяснимость, но дорого) или только решения и эффекты? Для latency-critical (<100 мс) durable-on-every-step нежелателен.

Повтор с отсрочкой временный сбой (таймаут API, сеть) недоведение задачи до конца
Таймаут шаг завис навсегда вечное зависание системы
Аренда два воркера взяли одну задачу двойная работа, расход токенов
Журнал событий падение на середине, частичный успех потеря состояния, невосстановимый отказ
Устойчивое исполнение гарантия доведения до конца через replay каждый сбой — потеря задачи
Недетерминизм LLM → replay эффектов классический replay ломается на LLM неверное восстановление

7. Режимы отказа — на что обратить внимание ОСНОВНОЕ

ОХВАТ ТАБЛИЦЫ Отказ · Чего не хватило · Класс MAST · Двойной побочный эффект (двойной платёж/запись после повтора) · ключ идемпотентности на эффекте и ещё 3

Двойной побочный эффект (двойной платёж/запись после повтора) ключ идемпотентности на эффекте task verification
Шторм повторов (синхронные повторы кладут downstream) отсрочка+jitter, предохранитель — (инфраструктурный)
Частичный успех «повис» (полузаписанное состояние) журнал событий + компенсация/resume specification
Silent interface drift (схема tool сменилась; API «200 OK», данных нет) verifier на результате + строгие контракты task verification (~21%)
Deadlock (две задачи ждут друг друга «без ошибки») таймауты + детекция зависимостей inter-agent (~37%)
Накопление ошибок (длинная цепочка тихо деградирует) промежуточные контрольные точки/проверки; см. урок 4 specification (~42%)
Потеря памяти/контекста после деплоя (stateless-дизайн) устойчивое состояние вне процесса —

Checklist: «система иногда падает» превращается в «между шагом N и N+1 нет verifier'а» или «эффект E неидемпотентен». Replit-инцидент читается как накопление ошибок + отсутствие журнала событий/компенсации.

8. Опорный стержень ОСНОВНОЕ

Наполняем Task и executor из урока 2 машиной состояний и контрактом устойчивости.

from enum import Enum

class Status(str, Enum):
    QUEUED = "queued"
    IN_PROGRESS = "in_progress"
    WAITING_APPROVAL = "waiting_approval"   # смычка с policy/HITL (уроки 1, 5, 12)
    SUCCEEDED = "succeeded"
    FAILED = "failed"
    TIMED_OUT = "timed_out"
    CANCELLED = "cancelled"
    ABORTED = "aborted"                      # критический системный сбой

# Таблица допустимых переходов — единственный источник правды.
# Любой переход вне таблицы = баг (тест в §9).
TRANSITIONS = {
    Status.QUEUED:           {Status.IN_PROGRESS, Status.CANCELLED},
    Status.IN_PROGRESS:      {Status.SUCCEEDED, Status.FAILED, Status.TIMED_OUT,
                              Status.WAITING_APPROVAL, Status.ABORTED},
    Status.WAITING_APPROVAL: {Status.IN_PROGRESS, Status.CANCELLED},
    Status.FAILED:           {Status.IN_PROGRESS, Status.ABORTED},   # retry по политике
    Status.TIMED_OUT:        {Status.IN_PROGRESS, Status.WAITING_APPROVAL, Status.ABORTED},
    # SUCCEEDED / CANCELLED / ABORTED — терминальные (нет исходящих переходов)
}

@dataclass
class Step:
    id: str
    idempotency_key: str        # покрывает ПОБОЧНЫЙ ЭФФЕКТ, не рассуждение модели (§4)
    cost: int                   # бюджет (planner-инвариант, урок 2)

@dataclass
class Event:                     # неизменяемая запись журнала событий (источник правды)
    ts: str
    task_id: str
    kind: str                   # task_created | tool_requested | tool_completed | status_changed ...
    idempotency_key: str | None
    payload_hash: str           # хэш входа/выхода, не сырьё (приватность + объём)

class Lifecycle(Protocol):
    def transition(self, task: Task, to: Status) -> Task: ...
        # инвариант: to ∈ TRANSITIONS[task.status], иначе IllegalTransition
    def append(self, e: Event) -> None: ...
        # инвариант: append-only; журнал — источник правды для replay
    def resume(self, task_id: str) -> Task: ...
        # реплеит журнал событий → восстанавливает состояние; идемпотентные шаги НЕ перевыполняются

Контракт для следующих этапов: любой компонент, выполняющий побочный эффект, обязан (1) проводить его через idempotency_key, (2) писать Event до и после эффекта, (3) объявить компенсацию для необратимых действий. Исполнитель (урок 2) теперь обязан возвращать tool-error как данные (не глотать) — иначе ломается переход в FAILED/повтор. Replit-инцидент = деструктивный эффект без идемпотентности, без Event, без компенсации.

9. Практикум ПРАКТИКУМ

Задача. Реализовать жизненный цикл: (а) запретить недопустимые переходы; (б) гарантировать, что повтор не дублирует побочный эффект; (в) восстановить состояние из журнала событий; (г) корректно обработать недетерминированный шаг.

Скелет (lifecycle.py):

class TaskRunner:
    def __init__(self, store, effects, *, max_retries=3, base_backoff_ms=100):
        self.store = store          # устойчивое хранилище: статусы + append-only журнал событий
        self.effects = effects      # реестр идемпотентных побочных эффектов

    def transition(self, task, to):
        if to not in TRANSITIONS.get(task.status, set()):
            raise IllegalTransition(task.status, to)
        self.store.append(Event(now(), task.id, "status_changed", None, hash(to)))
        task.status = to
        return task

    def run_step(self, task, step):
        # идемпотентность: если эффект с этим ключом уже зафиксирован — вернуть сохранённое
        if (cached := self.store.find_effect(step.idempotency_key)) is not None:
            return cached
        for attempt in range(self.max_retries):
            try:
                self.store.append(Event(now(), task.id, "tool_requested",
                                        step.idempotency_key, hash(step)))
                res = self.effects.apply(step)               # ЕДИНСТВЕННАЯ точка side-effect
                self.store.save_effect(step.idempotency_key, res)   # кэш результата
                self.store.append(Event(now(), task.id, "tool_completed",
                                        step.idempotency_key, hash(res)))
                return res
            except TransientError:
                sleep(self.base_backoff_ms * (2 ** attempt))  # exp backoff
        return self.transition(task, Status.FAILED)

    def resume(self, task_id):
        ...  # реплей журнала событий: восстановить статус и НЕ перевыполнять зафиксированные эффекты

Тесты:

def test_illegal_transition_rejected():
    with pytest.raises(IllegalTransition):
        runner.transition(Task(status=Status.SUCCEEDED), Status.IN_PROGRESS)  # терминальный

def test_retry_does_not_double_side_effect():
    # тот же idempotency_key дважды → эффект применён ОДИН раз (анти «двойной платёж»)
    r1 = runner.run_step(task, payment_step)
    r2 = runner.run_step(task, payment_step)
    assert r1 == r2
    assert effects.apply_count(payment_step.idempotency_key) == 1

def test_resume_reconstructs_state_without_replaying_effects():
    runner.run_step(task, step_a)          # зафиксирован
    crash()                                 # процесс «умер»
    recovered = TaskRunner(store, effects).resume(task.id)
    assert recovered.status == task.status
    assert effects.apply_count(step_a.idempotency_key) == 1   # НЕ выполнен повторно

def test_nondeterministic_step_is_safe_to_resume():
    # LLM-шаг даёт разный текст, но идемпотентность покрывает эффект, не выход
    runner.run_step(task, llm_step_with_effect)
    runner.resume(task.id)
    assert effects.apply_count(llm_step_with_effect.idempotency_key) == 1

Эталон. Ключевые мысли: (1) переходы валидны только по TRANSITIONS — машина состояний, а не «status = любая строка»; (2) идемпотентность покрывает эффект, поэтому повтор и возобновление безопасны даже при недетерминированном LLM (test_nondeterministic... — суть §4); (3) replay восстанавливает из журнала, не перевыполняя зафиксированное. test_retry_does_not_double — буквально анти-паттерн двойного платежа.

Расширение. Добавь compensate(step) для необратимых эффектов и тест: при ABORTED после частичного успеха выполняются компенсации в обратном порядке (паттерн сага (saga)).

11. Глоссарий СПРАВКА
  • Машина состояний задачи — конечный набор статусов + таблица допустимых переходов; единственный источник правды о состоянии. Переход вне таблицы = баг.
  • Идемпотентность — повтор операции с тем же ключом не создаёт новый побочный эффект. Ключ покрывает эффект, не рассуждение модели.
  • Building on Quicksand (Helland & Campbell, 2009) — канон: для длинных распределённых активностей exactly-once недостижим → at-least-once + идемпотентность + компенсации/«apologies».
  • At-least-once + идемпотентность — практичная замена недостижимому exactly-once.
  • Повтор / экспоненциальная отсрочка / jitter — повтор временно сбойного шага с растущими интервалами; гасит шторм повторов.
  • Аренда — временная эксклюзивная аренда задачи воркером; истекает при падении → задача возвращается в очередь.
  • Журнал событий — неизменяемый append-only журнал произошедшего; источник правды для replay и аудита.
  • Устойчивое исполнение — гарантия доведения задачи до конца через авто-persistence + возобновление с точки сбоя (Temporal, Restate, Dapr Workflows).
  • Контрольная точка — снэпшот состояния между шагами (LangGraph). Не равна устойчивому исполнению (спор §5).
  • Частичный успех / компенсация (сага) — часть эффектов произошла; откат уже сделанного обратными операциями.
  • Предохранитель — при пороге ошибок к сервису временно прекратить запросы.
  • Накопление ошибок — экспоненциальная деградация длинной цепочки (p^N); мост к уроку 4.
  • Воспроизводимость (для агентов) — повтор траектории решений и эффектов из журнала, не токенов.
