Чему вы научитесь

После этого урока вы будете: - Понимать, почему права должны жить вне модели, и что значит принцип «agent proposes, system decides». - Строить движок политик (policy engine) с доверительным принудительным исполнением: белый список (allowlist), уровни риска, режимы действий, привязка к происхождению данных. - Воспроизводить тестами реальные инциденты (EchoLeak, Step Finance) и доказывать, что блокировки по умолчанию их останавливают. - Различать ролевое управление доступом (RBAC), атрибутное управление доступом (ABAC), учётные данные по требованию (JIT), делегированную авторизацию (OAuth 2.1/PKCE) и их место в архитектуре агента. - Аргументированно выбирать уровень автономии (ось B) и размещение принудительного исполнения в стеке.

0. Зачем этот урок КОНТЕКСТ

Урок 1 дал ось B (полномочия). Урок 2 ввёл движок политик (policy engine) и реестр инструментов (tool registry) как примитивы. Урок 4 пометил контент по trust. Здесь мы собираем из этого слой прав и политик (policy layer): как агент получает доступ к инструментам так, чтобы (а) делать работу и (б) не превращаться в катапульту при обходе ограничений (jailbreak) или внедрении в промпт (prompt injection). Это последний «инфраструктурный» этап перед evals (6); этап 10 займётся полной моделью угроз (threat model), а здесь — конкретно права, режимы действий и принудительное исполнение (enforcement).

Две мысли, которые урок должен оставить:

КЛЮЧЕВАЯ МЫСЛЬ

(1) Уровень автономии (ось B) реализуется слоем политик вне модели: agent предлагает действие, система решает, исполнять ли. Права, которые живут только в промпте, не существуют — их снимает первая инъекция. (2) Опасность инструмента — не в его «деструктивности», а в комбинации прав, источника входных данных (trust) и отсутствия гейтов. Read-only инструмент тоже эксфильтрует данные, если ему скормить недоверенный ввод.

1. Конкретная проблема: один человек, 195 миллионов записей

Февраль 2026. Один атакующий убедил Claude действовать как «элитный хакер» (jailbreak), и с помощью агента за меньше месяца взломал 9 правительственных агентств Мексики, эксфильтровав ~195 млн записей (~150 ГБ) — налоговые и гражданские данные. Атака шла тысячами команд через десятки сессий; значительная часть шагов выполнялась агентом самостоятельно. Источники: LA Times, 2026-02-26; SeComPass analysis.

Что позволяло агенту это делать?

agent = Agent(tools=[shell, db, http, filesystem, email])  # «пусть умеет всё»
agent.run(user_input)                                       # любое user_input → любой tool

Это буквально конфигурация случившегося. Агент имел широкий доступ, никаких ограничений на частоту, никакого мониторинга объёма, никакого внешнего контроля. Права жили в доверии к модели — обход ограничений их просто снял.

Отсюда семь пробоин, каждая со своим решением. Разберём их по порядку — каждый раз от конкретной проблемы к простому коду.

2. Первый примитив: белый список (allowlist) — «агент мог вызывать что угодно»

В разборе инцидента агент имел доступ ко всем инструментам — и к тем, которые не были нужны для задачи. shell.exec для выкачивания данных, db.query для массовой выборки, http.fetch для эксфильтрации.

Что происходит

Наивная конфигурация перечисляет все инструменты агенту. Как только обход ограничений снят — они все доступны атакующему.

Решение: белый список

Вместо «всего, кроме» — «только это»:

allowed_tools = {"db.read", "ticket.search", "kb.query"}   # явное разрешение
all_tools = {"shell.exec", "db.read", "http.fetch", ...}

for t in all_tools:
    if t.name in allowed_tools:
        registry.register(t)       # агент ВИДИТ только разрешённые
    # else: инструмент просто не существует для агента

Что даёт белый список

Атакующий не может вызвать http.fetch, потому что агент о нём «не знает». Даже с полным jailbreak — список инструментов ограничен. Это операционализация минимальных привилегий (least privilege): всё запрещено, пока явно не выдано.

Цена белого списка

Но вот тонкость: db.read — тоже опасен, если через него выгрузить 195M записей. Белый список сам по себе не различает как именно агент использует разрешённый инструмент. Нужно классифицировать сами инструменты по степени риска.

3. Второй примитив: уровни риска (risk tiers) — «не все инструменты равноопасны»

В Мексике атакующий выгружал данные через легитимный db.query. Инструмент был в белом списке — проблема не в доступе, а в том, что частотный мониторинг и риск-классификация отсутствовали.

Решение: уровни риска

Классифицируем каждый инструмент по потенциальному вреду:

class Risk(IntEnum):
    LOW = 0       # read-only запрос, возвращает одно значение
    MEDIUM = 1    # запись в изолированную область
    HIGH = 2      # запись в общую базу, shell-команда
    CRITICAL = 3  # перевод средств, массовое удаление

db_query = ToolSpec(name="db.query", risk=Risk.LOW)
shell_exec = ToolSpec(name="shell.exec", risk=Risk.HIGH)
payment = ToolSpec(name="payment.transfer", risk=Risk.CRITICAL)

Что дают уровни риска

Теперь низкорисковый db.query может работать автоматически, а высокорисковый shell.exec требует проверки — ещё до того, как конкретный запрос analysed.

Что не решено

Но как именно «требует проверки»? Должен ли агент ждать человека на каждом вызове shell.exec? Очевидно, нет — некоторые вызовы модели разрешены, некоторые требуют человека. Нужен следующий слой.

4. Третий примитив: режимы действий (action modes) — «деструктив требует человека»

Step Finance (январь 2026): у торговых ботов было право двигать средства без человеческой проверки; при компрометации это вылилось в кражу 261,854 SOL (~$27M), после чего платформа закрылась.

Проблема: деструктивное действие было разрешено в автоматическом режиме.

Решение: режимы действий

Продолжение оси B (урок 1). Каждый инструмент работает в одном из режимов:

class Decision(str, Enum):
    AUTO = "auto"           # исполняется без человека
    APPROVAL = "approval"   # требует одобрения (человек в цикле)
    BLOCK = "block"         # запрещено всегда

def decide(tool_spec) -> Decision:
    if tool_spec.destructive:
        return Decision.APPROVAL    # деструктив НИКОГДА не AUTO
    if tool_spec.risk == Risk.LOW:
        return Decision.AUTO
    return Decision.APPROVAL

Что дают режимы действий

Деструктивные операции (перевод средств, DROP TABLE) никогда не проходят в автоматическом режиме. Step Finance был бы остановлен — человек увидел бы запрос на одобрение.

Ограничение

Но все эти решения живут в коде агента. Что если атакующий убедит модель «сказать» системе, что одобрение получено? Права в промпте не живут — следующая, критическая пробоина.

5. Четвёртый примитив: внешнее принудительное исполнение — «права в промпте не работают»

EchoLeak (CVE-2025-32711, CVSS 9.3): zero-click внедрение в промпт в M365 Copilot. Письмо со скрытыми инструкциями — при суммаризации Copilot слепо выполнил их и эксфильтровал файлы из OneDrive/SharePoint. Все «защиты» жили в промпте — и были сняты инъекцией.

Ключевой вывод: если политика живёт только в системном промпте («не удаляй прод-данные»), она ровно настолько сильна, насколько устойчив промпт к инъекции. А инъекция, как показал EchoLeak, снимает защиту целиком.

Решение: принудительное исполнение вне модели

Движок политик — отдельный слой, который модель не контролирует:

class PolicyEngine:
    def decide(self, req: ActionRequest, spec: ToolSpec) -> Decision:
        # Модель не вызывает decide. Это слой вне её кода.
        if not spec.allowed:                              return Decision.BLOCK
        if spec.destructive:                              return Decision.APPROVAL
        if spec.risk > req.max_autonomy:                  return Decision.APPROVAL
        return Decision.AUTO

Модель предлагает вызов инструмента. Исполнитель (executor, урок 2) перехватывает вызов и пропускает через PolicyEngine.decide. Даже если модель «сказала», что действие одобрено — движок политик проверяет независимо. Принцип: agent proposes, system decides.

Цена внешнего принудительного исполнения

Латентность (дополнительный вызов движка) и сложность интеграции. Но альтернатива — права, которые исчезают при инъекции.

Новая проблема

Вернёмся к EchoLeak: Copilot обрабатывал недоверенное письмо. Инструмент fetch_file был разрешён, не деструктивен — и всё равно эксфильтровал данные. Потому что политика не смотрела на происхождение аргументов.

6. Пятый примитив: политика с учётом происхождения — «недоверенный ввод меняет уровень гейта»

В Мексике и EchoLeak одна и та же архитектурная проблема: легитимный инструмент исполняет вредоносный интент из недоверенного источника — классическая проблема спутанного заместителя (confused deputy). Инструмент не делает ничего плохого по своей спецификации, но данные, которые он обрабатывает, отравлены.

Решение: политика зависит от trust аргументов

Урок 4 ввёл arg_trust — verified или untrusted. Теперь связываем это с политикой:

def decide(req: ActionRequest, spec: ToolSpec) -> Decision:
    if not spec.allowed:                                          return Decision.BLOCK
    if req.arg_trust == "untrusted" and spec.risk >= Risk.MEDIUM:
        return Decision.APPROVAL    # недоверенный ввод — нужен человек
    return Decision.AUTO

http.fetch разрешён? Да. Недоверенный ввод? Да. Значит — не AUTO, требуется человек. EchoLeak блокируется на этом уровне: даже fetch легитимного файла требует одобрения, если инструкция пришла из письма.

Что даёт trust-aware политика

Связка двух уроков: trust (4) + политика (5). Читать файл можно, но читать файл по инструкции из недоверенного ввода — нет.

Остаётся

Атакующий всё ещё может действовать внутри сессии с доверенными правами. Для этого нужны эфемерные, одноразовые ключи — чтобы каждое действие авторизовалось отдельно.

7. Шестой примитив: учётные данные по требованию (JIT) — «украли долгоживущий ключ»

Step Finance использовала общие долгоживущие ключи для всех торговых ботов. Компрометация одного → доступ ко всем средствам.

Решение: эфемерные токены под одну операцию

def execute_with_jit(tool: str, args: dict, scope: str) -> Result:
    token = auth.request_token(
        scope=scope,              # ровно этот инструмент + эти аргументы
        ttl=30,                   # живёт 30 секунд
        one_shot=True             # одно использование
    )
    result = tool.run(args, credential=token)
    token.revoke()                # сразу отозван
    return result

Даже если токен перехвачен — он невалиден через 30 секунд, не работает для других инструментов, нельзя использовать дважды. Противоположность «одному ключу на всё».

8. Седьмой примитив: журнал аудита — «нельзя разобрать инцидент» ПРИМЕР

После Мексики следователи не могли восстановить последовательность: какие команды отдавал атакующий, какие инструменты вызывались, какие данные были затронуты. Аудита не было.

Решение: неизменяемый лог каждого решения

def decide(req: ActionRequest, spec: ToolSpec) -> Decision:
    decision = _decide(req, spec)
    audit.record({
        "tool": req.tool, "args": req.args, "arg_trust": req.arg_trust,
        "decision": decision, "time": now(), "session": session_id
    })
    return decision

Любое решение (AUTO, APPROVAL, BLOCK) записывается. После инцидента есть цепочка: «в 14:03:22 агент предложил db.query со всеми записями; политика вернула AUTO (risk был LOW, лимита частоты не было)». Смычка с event log урока 3.

Семь примитивов построены. Теперь сведём их в единую картину.

9. Сводка: карта решений ИНСТРУМЕНТ

Теперь, когда каждый примитив построен от конкретной пробоины, сведём их в таблицу для выбора архитектуры:

ОХВАТ ТАБЛИЦЫ Ось · Варианты · Сила · Цена · Allowlist и ещё 3
Ось Варианты Сила Цена
Allowlist белый список vs чёрный список минимальные привилегии по умолчанию узкий список калечит полезность

Принцип «agent proposes, system decides» связывает всё: модель предлагает, внешний слой политик решает auto/approval/block. Ни один слой не достаточен — нужна эшелонированная защита (defense-in-depth): allowlist + risk tiers + action modes + trust-aware + JIT + audit.

10. Режимы отказа — короткий чеклист ОСНОВНОЕ

ОХВАТ ТАБЛИЦЫ Отказ · Механизм · Противоядие · Privilege abuse (Мексика) · агент имеет лишние права, обход ограничений снимает все гейты и ещё 2
Отказ Механизм Противоядие
Privilege abuse (Мексика) агент имеет лишние права, обход ограничений снимает все гейты allowlist + risk tiers + JIT
Injection → tool misuse (EchoLeak) инъекция снимает защиту в промпте внешний policy engine
Деструктив без HITL (Step Finance) автономное исполнение необратимого действия режим APPROVAL для destructive
Confused deputy легитимный инструмент исполняет чужой интент trust-aware policy
Кража ключей долгоживущий ключ скомпрометирован JIT-токены
Нет расследования нет лога, нельзя восстановить инцидент audit trail

Практический смысл: «агента взломали» → адресный диагноз «деструктив был в AUTO» или «политика не смотрела на trust аргументов».

11. Опорный стержень ОСНОВНОЕ

Наполняем PolicyEngine и ToolRegistry из урока 2, связывая с осью B (урок 1) и trust (урок 4).

from enum import IntEnum

class Risk(IntEnum):
    LOW = 0; MEDIUM = 1; HIGH = 2; CRITICAL = 3

class Decision(str, Enum):
    AUTO = "auto"; APPROVAL = "approval"; BLOCK = "block"

@dataclass
class ToolSpec:                      # запись tool_registry
    name: str
    risk: Risk
    destructive: bool
    allowed: bool                    # белый список: по умолчанию False (минимальные привилегии)
    schema: dict                     # JSON-schema аргументов (контракт)

@dataclass
class ActionRequest:
    tool: str
    args: dict
    arg_trust: str                   # verified | untrusted  (из урока 4!)
    max_autonomy: Risk               # потолок оси B для этой задачи (урок 1, Task.constraints)

class PolicyEngine(Protocol):
    def decide(self, req: ActionRequest, spec: ToolSpec) -> Decision: ...
        # ИНВАРИАНТЫ:
        #  (1) not spec.allowed                      -> BLOCK         (белый список)
        #  (2) spec.destructive                      -> APPROVAL/BLOCK (никогда AUTO)
        #  (3) spec.risk > req.max_autonomy          -> APPROVAL       (ось B урока 1)
        #  (4) req.arg_trust == "untrusted" и risk>=MEDIUM -> APPROVAL (анти-EchoLeak)
        #  (5) ЛЮБОЕ решение -> запись в журнал аудита (урок 3 event log)
        # Принудительное исполнение ВНЕ модели: модель не вызывает decide за себя.

Матрица политик (артефакт, который наполняет матрицу):

ОХВАТ ТАБЛИЦЫ Действие · Risk · Гейт · Лог · Откат и ещё 1
Действие Risk Гейт Лог Откат
db.query (read) LOW auto (если в белом списке) да n/a
file.write (в allowed path) MEDIUM auto; вне path → approval да версионирование
db.write HIGH всегда approval да транзакция
shell.exec HIGH approval; деструктив → human да snapshot
payment.transfer CRITICAL pre-action approval + лимит да ручная компенсация

Контракт для следующих этапов: исполнитель (урок 2) обязан вызывать policy.decide перед каждым побочным эффектом; context-builder (урок 4) передаёт arg_trust; verifier (этап 6) тестирует, что политика реально блокирует; человек в цикле (этап 12) — канал для APPROVAL. Разбор инцидента = нарушение инвариантов (1)+(3).

12. Практикум — движок политик с доверительным принудительным исполнением ПРАКТИКУМ

Задача. Реализовать decide по инвариантам §11 и tool-filter на входе реестра. Доказать тестами, что воспроизводятся реальные инциденты: деструктив без человека блокируется; внедрение через недоверенный ввод требует одобрения даже для разрешённого инструмента.

Скелет (policy.py, артефакты — реальный OpenAI Agents SDK API):

# 1) Фильтр инструментов на загрузке (белый список) — реальный SDK
filesystem_server = MCPServerStdio(
    params={"command": "npx",
            "args": ["-y", "@modelcontextprotocol/server-filesystem", "/safe/dir"]},
    tool_filter=create_static_tool_filter(allowed_tool_names=["read_file", "write_file"]),
)  # delete_file недоступен агенту в принципе

# 2) Runtime движок политик
def decide(req: ActionRequest, spec: ToolSpec) -> Decision:
    audit.record(req, spec)                                  # инвариант (5): лог всегда
    if not spec.allowed:                       return Decision.BLOCK        # (1)
    if spec.destructive:                       return Decision.APPROVAL     # (2)
    if spec.risk > req.max_autonomy:           return Decision.APPROVAL     # (3)
    if req.arg_trust == "untrusted" and spec.risk >= Risk.MEDIUM:
        return Decision.APPROVAL                                            # (4) анти-EchoLeak
    return Decision.AUTO

Тесты:

def test_unlisted_tool_blocked():
    assert decide(req("delete_file"), spec(allowed=False)) == Decision.BLOCK

def test_destructive_never_auto():            # анти Step Finance
    assert decide(req("payment.transfer", max_autonomy=Risk.CRITICAL),
                  spec(risk=Risk.CRITICAL, destructive=True)) == Decision.APPROVAL

def test_risk_above_autonomy_ceiling_requires_approval():   # ось B урока 1
    assert decide(req("db.write", max_autonomy=Risk.LOW),
                  spec(risk=Risk.HIGH)) == Decision.APPROVAL

def test_untrusted_args_escalate_even_for_allowed_tool():   # анти EchoLeak / спутанный заместитель
    # инструмент разрешён, не деструктивен, risk НЕ выше потолка — изолируем именно инвариант (4)
    assert decide(req("http.fetch", arg_trust="untrusted", max_autonomy=Risk.HIGH),
                  spec(risk=Risk.MEDIUM, allowed=True, destructive=False)) == Decision.APPROVAL

def test_every_decision_is_audited():
    decide(req("db.query"), spec(risk=Risk.LOW, allowed=True))
    assert audit.last() is not None                          # инвариант (5)

def test_prompt_cannot_self_authorize():
    # модель "сказала" что действие одобрено — это не влияет на decide (принудительное исполнение вне модели)
    req_with_fake_approval = req("shell.exec", model_says_ok=True)
    assert decide(req_with_fake_approval, spec(risk=Risk.HIGH)) == Decision.APPROVAL

Эталон. Главные мысли: (1) минимальные привилегии — дефолт BLOCK; (2) test_untrusted_args... — связывает trust урока 4 с политикой: разрешённый, недеструктивный инструмент всё равно требует одобрения, если ввод недоверенный (это и есть защита от EchoLeak-класса); (3) test_prompt_cannot_self_authorize — буквально «agent proposes, system decides»: слова модели не меняют решение. test_destructive_never_auto — анти-Step-Finance.

Расширение. Добавь учётные данные по требованию: decide для APPROVAL-действия выдаёт эфемерный токен с временем жизни (TTL) и областью действия (scope) под одну операцию; тест — токен невалиден после использования и вне области действия.

13. Вопросы на суждение ПРОВЕРКА

  1. Команда «защитила» агента, прописав в системном промпте «никогда не удаляй прод-данные». Почему по §1 (EchoLeak, Мексика) это не защита? Где должно жить правило?
  2. Read-only инструмент считается безопасным. Приведи сценарий (по EchoLeak), где он приводит к эксфильтрации. Что в политике это ловит (§11, инвариант 4)?
  3. Дан агент с правом перевода средств в limited-auto «для скорости». По Step Finance — какой инвариант §5 нарушен и какой компромисс (скорость vs радиус поражения) тут делают неверно?
  4. Когда белый список лучше чёрного и когда узкий белый список становится проблемой? Как решить, дробить ли монолитного агента на роли?
  5. Почему «agent proposes, system decides» требует, чтобы принудительное исполнение было вне модели? Что именно ломается, если решение auto/block принимает сам LLM?
  6. Open question: спроектируй процесс владения и тестирования политик (кто пишет, как версионируют, как тестируют как код). Где твой процесс отстанет от скорости добавления новых инструментов?

14. Глоссарий СПРАВКА
  • Белый список / Чёрный список — явный позитивный (разрешено только это) / негативный список. Белый список = минимальные привилегии по умолчанию.
  • Минимальные привилегии — минимально необходимые права; всё запрещено, пока явно не выдано.
  • Уровень риска — классификация действия по потенциальному вреду (низкий→критический); определяет строгость гейта.
  • Режимы действий — только чтение / черновик / запись с одобрением / ограниченный автомат / деструктивное требует человека; реализация оси B урока 1.
  • Политика как код — правила в исполняемой форме (Rego, Cedar, Cerbos), а не прозой в промпте; машинное принудительное исполнение + аудит.
  • «Agent proposes, system decides» — модель предлагает действие, внешний слой политик решает auto/approval/block. Принудительное исполнение вне модели.
  • Точка принятия решений — внешний движок решения (OPA/Cedar), отделённый от модели.
  • Учётные данные по требованию — эфемерные токены под одну операцию с временем жизни и областью действия вместо долгоживущих общих ключей.
  • Журнал аудита — неизменяемый лог действий/решений (смычка с event log урока 3).
  • Проблема спутанного заместителя — легитимный инструмент исполняет вредоносный интент из недоверенного источника; противоядие — политика с учётом trust.
  • Смертельная триада (Уиллисон) — опасная триада private data + untrusted content + external communication; безопасны ≤2 из трёх. Предшественник Правила двух от Meta (урок 10).
  • OAuth 2.1 / PKCE / от имени пользователя — делегированная авторизация агента (спецификация MCP); ограниченный доступ от имени пользователя.
  • Нулевое доверие / аварийный выключатель — никакого неявного доверия; мгновенное отключение агента (отзыв токенов/сети).
  • Ролевое управление доступом / Атрибутное управление доступом — права по роли и по динамическим атрибутам (время, чувствительность данных).
