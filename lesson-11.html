0. Зачем этот урок

Десять этапов мы строили одного агента-оркестратора. Теперь вопрос: когда нужно несколько агентов, как они координируются, не мешая друг другу, и как агенты от разных вендоров вообще разговаривают. Это усиленный этап: после него у нас есть критерий «сколько агентов», протокол передачи и — главное — правильное (не перепутанное) понимание MCP vs A2A.

Две мысли, которые урок должен оставить:

(1) «Больше агентов» — не бесплатно и не автоматически лучше. Мультиагент выигрывает на распараллеливаемой широте (Anthropic: +90.2% над одиночным на research), но добавляет целый класс отказов — координацию (MAST: 37% отказов MAS — рассогласование агентов) и платит ~15× токенов. 40% agentic-проектов отменят к 2027 (Gartner). Сначала спроси «нужен ли вообще второй агент». (2) Два протокола решают разные задачи и дополняют друг друга: MCP — вертикаль (агент↔инструмент/данные, Anthropic), A2A — горизонталь (агент↔агент, Google→Linux Foundation). Это не конкуренты; путать их — типовая ошибка.

1. Cold-open: мультиагент, который выиграл +90% и сжёг 15× — и тот, что породил рой

Мультиагентность продают как «просто добавь агентов — будет умнее». Реальность двойственна, и обе стороны верифицированы:

  • Выигрывает (иногда): Anthropic («How we built our multi-agent research system») — связка Opus-4-lead + Sonnet-4-субагенты обошла одиночного Opus 4 на +90.2% на внутреннем research-eval. Для распараллеливаемой широты (искать по многим направлениям) мультиагент реально лучше.
  • Платит дорого: та же система — ≈15× токенов против чата. Координация не бесплатна.
  • Добавляет новый класс отказов: MAST (arXiv 2503.13657, верифицировано в уроках 2–3) — 37% отказов MAS — это inter-agent misalignment: агенты не передают критичное, расходятся в целях. Координация сама — крупный источник сбоев.
  • Часто не окупается: Gartner — >40% agentic-AI проектов отменят к 2027 (стоимость, неясная ценность, риски).
  • Может выродиться в рой: ранний провал Anthropic (урок 2) — лид породил 50 субагентов на простой запрос, решаемый одним.

Отсюда тезис урока:

Мультиагентность — это инструмент с резко нелинейным профилем: на правильной задаче (параллельная широта) она даёт большой выигрыш, но вносит новый класс отказов (координация, 37%) и кратную стоимость (15×). Дефолт — один агент с инструментами; несколько агентов вводят только когда задача реально требует параллелизма или разделения границ ответственности/безопасности.

(Карантин: «200мс→4с латентность» (4-й рецидив фабрикации), «41–86.7% / 1642 прогона», «42.68% vs 2.92%», «79% используют/96% расширяют» — не подтверждены; беру верифицированные Anthropic/MAST/Gartner. §9.)

2. Деривация: от одного агента к скоординированной сети

2.1 Боль → механизм

  • «Нужен ли вообще второй агент?» Дефолт — single-agent + tools (проще, дешевле, меньше точек отказа). Мультиагент оправдан при: параллельной широте (Anthropic research), разделении зон безопасности (каждый агент — минимум прав, урок 10), разных командах-владельцах. Иначе — это «рой» (cold-open) и 15× счёт зря.
  • «Агенты дублируют работу» (50 субагентов). Вводим роли с границами: паттерн Coordinator–Implementor–Verifier (координатор декомпозирует, исполнители делают, верификатор проверяет — урок 6). Role boundaries устраняют дублирование. Конкретная реализация — Magentic-One (Microsoft): Orchestrator + специалисты. Отдельный паттерн — multiagent debate (Du et al.): несколько моделей спорят и сходятся к ответу (консенсус ради качества, не декомпозиция).
  • «Как передать задачу/контекст». Вводим handoff-протокол: триажный агент → специалист (sales / refund / human). Явная схема «кто что делает и кому передаёт».
  • «Общий контекст рассинхронизировался» (context drift). Один агент зовёт f(), другой уже переименовал её в g(). Вводим выбор shared vs private context + синхронизацию; для кода — изоляцию (git worktree), чтобы не было silent overwrite (два агента стёрли правки друг друга).
  • «Агенты дали противоречивые решения». Вводим conflict resolution (не молчаливый overwrite — урок 7!): арбитр, голосование, или приоритет по trust/score. Сложное — на человека (урок 12).
  • «Агенты от разных вендоров не понимают друг друга». Вводим inter-agent протоколы — §2.2.

2.2 MCP vs A2A: вертикаль и горизонталь (не путать)

Это центральный и самый путаемый пункт. Два протокола, разные оси, дополняют друг друга:

MCP (Model Context Protocol) A2A (Agent2Agent)
Ось вертикаль: агент ↔ инструмент/данные горизонталь: агент ↔ агент
Кто Anthropic (2024), modelcontextprotocol.io Google → Linux Foundation, v1.0 2026, 150+ орг
Задача подключить агента к tools/БД/сервисам (JSON-схемы tool-вызовов) обнаружение, делегирование, обмен между агентами
Ключевой артефакт tool schema (урок 5) AgentCard (визитка агента: name, url, capabilities), Signed Agent Cards (крипто-идентичность)

Аналогия: MCP — как агент берёт инструмент в руку; A2A — как два агента жмут руки. Они не конкурируют: агент использует MCP, чтобы дотянуться до своих инструментов, и A2A, чтобы делегировать задачу другому агенту. Путать их (что GPT делает часто) — значит не понимать архитектуру сети агентов.

MCP и A2A — два доминирующих, но ландшафт фрагментируется: есть ещё ACP (IBM/BeeAI), ANP и др.; систематизация — «A Survey of AI Agent Protocols» (Yang et al., 2504.16736). Тренд к стандартизации идёт, но не устаканился.

2.3 Новая проблема: мультиагент расширяет threat model (мост к уроку 10)

Каждый новый агент — новая поверхность атаки. Rogue agent (скомпрометированный агент заражает соседей через shared context/сообщения), перехват протокола, каскадное распространение poisoning (урок 7). Поэтому Rule of Two (урок 10) применяется к КАЖДОМУ субагенту, а доверие между агентами — через Signed Agent Cards. Мультиагентность не отменяет безопасность — она её умножает.

2.4 Новая проблема (мост к §3): протоколы незрелы, польза спорна

MCP/A2A — 2024–2026, без долгих полевых испытаний; реализации расходятся. И фундаментально: «мультиагент vs один агент + tools» не решён (Gartner 40% отмен). Это §3.

3. Карта решений: сколько агентов и как они говорят

Ось Варианты Когда Цена
Кол-во single-agent+tools vs multi-agent один: линейные/короткие; много: параллельная широта, разделение безопасности каждый переход = промпт + латентность + токены (15×)
Паттерн sequential / parallel / hierarchical (CIV) / dynamic pipeline предсказуем; parallel быстр; hierarchical контролируем; dynamic гибок координатор = единая точка отказа; parallel без синхр. → конфликты
Контекст shared vs private shared: общая картина; private: изоляция, нет drift shared → context drift, silent overwrite
Протокол MCP (tools) и A2A (agents) не «или-или»: MCP — вертикаль, A2A — горизонталь незрелы (2024–2026), реализации расходятся

3.1 Спорные точки (оставляем спорными)

  • Польза мультиагентности vs один агент — настоящий дебат. За: Anthropic «How we built our multi-agent research system» (+90.2% на research, за 15×). Против: Cognition «Don't Build Multi-Agents» — дробление контекста между агентами теряет общую картину и плодит рассогласование; для многих задач один агент с дисциплиной контекста надёжнее. Плюс координация — 37% отказов (MAST), 40% проектов отменят (Gartner). Практический синтез (LangChain): начинать с одного агента, вводить мультиагентность только под реальный параллелизм/разделение зон.
  • «Больше агентов» вредит. Накладные (context switching, синхронизация) могут перевесить параллелизм. Для однофайловых/простых задач один LLM лучше CIV.
  • Зрелость протоколов. MCP/A2A молоды, без долгих полевых испытаний, версии и multi-tenancy ещё дорабатываются. Между реализациями интерфейс может расходиться.
  • Метрики мультиагентности. Нет стандарта: критерий успеха локален (верный финал), системных метрик (частота конфликтов, эффективность обмена) в стандарте нет.

6. Pset 11 — координатор с границами ролей, изоляцией и Rule of Two

Задача. Реализовать координатор, который (а) не плодит агентов на простой задаче; (б) задаёт непересекающиеся роли; (в) изолирует контекст (анти silent-overwrite); (г) разрешает конфликт без overwrite; (д) применяет Rule of Two к субагентам.

Скелет (coordinator.py):

def should_spawn_multiagent(task) -> bool:
    # мультиагент стоит ~15× токенов (Anthropic) — оправдан только при параллелизме/разделении
    return task.is_parallelizable or task.needs_security_separation

def assign_roles(task):
    roles = decompose(task)                       # CIV: coordinator/implementor/verifier
    assert no_overlap(roles)                      # role boundaries (анти-дублирование)
    return roles

def run(task):
    if not should_spawn_multiagent(task):
        return single_agent.run(task)             # дефолт: один агент (анти-рой)
    bindings = assign_roles(task)
    for b in bindings:
        assert rule_of_two_ok(b.agent.caps)       # урок 10 на каждого субагента
    results = parallel_run(bindings, context_mode="private")   # изоляция (анти overwrite/drift)
    return resolve_conflict(collect(results))

def resolve_conflict(decisions):
    if consistent(decisions): return merge(decisions)
    return arbiter(decisions)                     # trust-приоритет/голосование, НЕ overwrite (урок 7)

Тесты:

def test_simple_task_uses_single_agent():
    assert should_spawn_multiagent(simple_faq_task) is False     # анти-рой (cold-open)

def test_parallelizable_task_spawns_multi():
    assert should_spawn_multiagent(broad_research_task) is True

def test_roles_do_not_overlap():
    roles = assign_roles(task)
    assert no_overlap(roles)                                     # анти-дублирование (50 субагентов)

def test_private_context_prevents_silent_overwrite():
    out = run(two_agents_editing_same_file)
    assert out.no_lost_edits                                     # изоляция (git worktree)

def test_conflict_not_silently_overwritten():
    res = resolve_conflict([decision_A_high_trust, decision_B_low_trust])
    assert res.method in ("arbiter", "vote", "trust_priority")   # не overwrite (урок 7)

def test_subagent_obeys_rule_of_two():
    sub = SubAgent(caps=all_three_capabilities())                # урок 10
    assert not passes_security(sub)                              # rogue-профиль блокируется

Эталон. Главные мысли: (1) test_simple_task_uses_single_agent — дефолт один агент, мультиагент только по необходимости (cold-open «рой» + 15× стоимость); (2) role boundaries против дублирования; (3) private context против silent overwrite/drift; (4) конфликт — арбитром, не overwrite (урок 7); (5) Rule of Two на каждого субагента (урок 10) — мультиагентность не отменяет безопасность.

Расширение. Добавь A2A-слой: AgentCard с подписью; делегирование задачи внешнему агенту проходит, только если его Signed Agent Card валиден (доверительная граница). Покажи тестом, что неподписанный агент в делегирование не допускается (анти rogue-agent).

7. Диагностические вопросы (проверяют суждение)

  1. PM требует «сделать мультиагентную систему, это современнее». По §1 (Gartner, MAST, 15×) — какие три вопроса зададут, прежде чем плодить агентов? Когда честный ответ — «хватит одного агента с ролями»?
  2. Коллега говорит: «используем A2A, чтобы агент дотянулся до базы данных». Что не так с этой фразой (§2.2)? Какой протокол для чего?
  3. Два агента параллельно правят один файл, и правки одного исчезли. Какой failure mode (§4) и какие два механизма (§2.1, §5) это предотвращают?
  4. Мультиагент дал +90.2% у Anthropic, но 37% отказов MAS — координация. Как примирить эти факты — когда разделение труда окупается, а когда вносит больше отказов, чем пользы (§3.1)?
  5. Вы добавили субагента, читающего внешние письма и имеющего доступ к внутренней БД и почте. Почему это нарушает Rule of Two (урок 10) на уровне субагента и чем грозит в мультиагентной сети (rogue)?
  6. Open question (§3.1): предложи метрику, отвечающую «окупилась ли мультиагентность на этой задаче» (учти качество, токены, частоту конфликтов). Чем её можно заиграть?

8. Глоссарий-дельта

  • Multi-agent vs single-agent — сеть специализированных агентов vs один агент с инструментами. Дефолт — один; мультиагент при параллелизме/разделении зон.
  • MCP (Model Context Protocol) — вертикаль: агент↔инструмент/данные (Anthropic, 2024). Не путать с A2A.
  • A2A (Agent2Agent) — горизонталь: агент↔агент (Google→Linux Foundation, v1.0 2026, 150+ орг). Обнаружение, делегирование, обмен.
  • AgentCard / Signed Agent Card — JSON-визитка агента (name/url/capabilities) / её крипто-подписанная версия для доверия (A2A).
  • CIV (Coordinator–Implementor–Verifier) — ролевой паттерн: декомпозиция / исполнение / проверка.
  • Role boundaries — непересекающиеся зоны ответственности; против дублирования (рой).
  • Handoff / delegation — передача задачи/контекста между агентами.
  • Shared vs private context — общий пул vs изоляция; shared → context drift / silent overwrite.
  • Context drift — рассинхрон представлений агентов (один не знает об изменениях другого).
  • Silent overwrite — параллельные правки стирают друг друга (лечится изоляцией, git worktree).
  • Rogue agent — скомпрометированный агент заражает сеть; противоядие — Rule of Two + Signed Cards (урок 10).
  • Arbiter / consensus — разрешение конфликта решений (голосование/trust-приоритет), не overwrite (урок 7).
